# Correlation & Trace Strategy вҖ” Module 3 Closure

**Branch:** `feature/payment-foundation`  
**Status:** Design documentation вҖ” no implementation  
**Applies to:** All future financial operations (Payment Engine, Event Bus, Audit, Webhooks)

---

## 1. Purpose

Every financial operation must be **traceable end-to-end** across:

- HTTP requests and retries
- Idempotency replays
- Transaction state changes
- Provider calls and webhooks
- Wallet movements
- Audit timeline reconstruction

This document defines who creates, owns, propagates, and mutates each identifier.

---

## 2. Identifier Definitions

| Identifier | Scope | Purpose |
|------------|-------|---------|
| **traceId** | Entire distributed trace (may span services, async jobs, webhooks) | Root correlation for observability (logs, APM, audit queries) |
| **correlationId** | Single business operation (e.g. one checkout attempt, one payout request) | Groups all records belonging to one logical financial action |
| **requestId** | Single inbound HTTP/API request | Distinguishes retries and concurrent requests within a correlation |
| **transactionId** | Platform payment transaction entity | Primary key for `payment_transactions` |
| **paymentReference** | Merchant/platform payment intent reference | Stable client-facing reference for a payment attempt |
| **providerReference** | Provider-side transaction ID | External PSP reference (MTN MoMo, Stripe, etc.) |

---

## 3. Ownership Matrix

| Identifier | Created by | Owned by | Stored in |
|------------|------------|----------|-----------|
| **traceId** | Edge gateway or first server handler (if absent, generated) | Platform (immutable once set) | Audit metadata, event envelope, logs, optional transaction metadata |
| **correlationId** | Client (header) or server (if absent) | Platform (immutable per business flow) | Idempotency record, audit record, transaction metadata, event envelope |
| **requestId** | Client (header) or server (per request) | Platform (immutable per request) | Idempotency record, audit record, transaction metadata, event envelope |
| **transactionId** | `TransactionService.createTransaction()` | Transaction domain | `payment_transactions.transactionId`, audit `resourceId` when resourceType=TRANSACTION |
| **paymentReference** | Client or Payment Engine at charge initiation | Payment domain | Transaction top-level field, idempotency context, audit metadata |
| **providerReference** | Provider adapter after successful provider call | Provider domain (platform stores copy) | Transaction top-level field, audit metadata, webhook payloads |

## 4. Creation Rules

### 4.1 traceId

```
IF client sends X-Trace-Id (or W3C traceparent)
  USE client value (normalized)
ELSE IF correlationId exists
  USE correlationId as traceId default
ELSE
  GENERATE crypto.randomUUID()
ENDIF
```

- **Created:** First platform touchpoint (middleware or Payment Engine entry)
- **Never changes** within a trace tree
- **Child async jobs** inherit parent traceId (webhooks, settlement workers)

### 4.2 correlationId

```
Source priority:
  1. X-Correlation-Id header
  2. req.correlationId (set by upstream middleware)
  3. req.idempotencyContext.correlationId
  4. crypto.randomUUID()  (server-generated)
```

- **Created:** IdempotencyMiddleware or Payment Engine (first handler)
- **Never changes** for the duration of one business operation
- **New checkout attempt** = new correlationId (even for same order if user retries manually)

**Relationship to idempotencyKey:**

| Field | Semantics |
|-------|-----------|
| `idempotencyKey` | "Have I already processed this exact submission?" |
| `correlationId` | "Show me everything that happened for this checkout attempt" |

One correlationId may map to one idempotencyKey. Retries with **same** idempotencyKey share correlationId. Retries with **new** idempotencyKey get new correlationId.

### 4.3 requestId

```
Source priority:
  1. X-Request-Id header
  2. req.idempotencyContext.requestId
  3. crypto.randomUUID()  (one per inbound HTTP request)
```

- **Created:** Per HTTP request (IdempotencyHelper.extractContext / AuditContext.fromRequest)
- **Never changes** within that request lifecycle
- **Each retry HTTP call** gets a new requestId (same or different correlationId depending on client)

### 4.4 transactionId

```
Created by: TransactionService.createTransaction()
Format: txn_<uuid>  (TransactionConfig prefix)
```

- **Created:** Once, inside idempotency handler (first successful claim)
- **Never changes**
- **Never reused** across idempotency replays вҖ” replay returns existing transactionId

### 4.5 paymentReference

```
Source priority:
  1. X-Payment-Reference header
  2. Client payload paymentReference
  3. Generated by Payment Engine: pay_<uuid>
```

- **Created:** At payment initiation (before provider call)
- **Never changes** for that payment attempt
- **Propagates** to provider metadata where supported

### 4.6 providerReference

```
Created by: Provider adapter after provider API response
Examples: MTN financialTransactionId, Stripe payment_intent id
```

- **Created:** When provider acknowledges transaction (may be null until then)
- **Set once** вҖ” corrections via new audit events, not field mutation
- **Updated in transaction record** only via `transitionStatus()` with provider data (append-style lifecycle field)

### 5.2 Propagation table

| Stage | traceId | correlationId | requestId | transactionId | paymentReference | providerReference |
|-------|---------|---------------|-----------|---------------|------------------|-------------------|
| HTTP ingress | set/propagate | set/propagate | set/new | вҖ” | optional set | вҖ” |
| Idempotency claim | pass | pass | pass | вҖ” | pass | вҖ” |
| Transaction create | metadata | metadata | metadata | **created** | stored | null |
| Audit PAYMENT_CREATED | metadata | record field | record field | resourceId | metadata | вҖ” |
| Provider charge | pass | pass | pass | pass | pass | **created** |
| Transaction CAPTURED | metadata | metadata | metadata | unchanged | unchanged | **set** |
| Event Bus publish | envelope | envelope | envelope | envelope | envelope | envelope |
| Audit subscriber | metadata | record field | record field | resourceId | metadata | metadata |
| Webhook ingress | inherit/new reqId | lookup by providerRef | new requestId | lookup | lookup | **match key** |

### 5.3 Webhook trace linking

Webhooks arrive without original HTTP headers. Linking strategy:

1. Extract `providerReference` from webhook payload
2. Lookup transaction by `providerReference` index
3. Inherit `correlationId`, `traceId`, `transactionId`, `paymentReference` from transaction metadata
4. Generate **new** `requestId` for the webhook HTTP request
5. Append audit events with inherited correlation + new requestId

## 6. Immutability Rules

| Identifier | Mutable? | Notes |
|------------|----------|-------|
| traceId | **Never** | Same value across sync + async legs |
| correlationId | **Never** | Per business operation |
| requestId | **Never** | Per HTTP request; new request = new requestId |
| transactionId | **Never** | Platform primary key |
| paymentReference | **Never** | After assignment at initiation |
| providerReference | **Set once** | Null вҶ’ value on first provider ACK; no overwrite with different value (new event if provider reissues) |

## 7. Storage Locations

| Store | Fields persisted |
|-------|------------------|
| `payment_idempotency_keys` | idempotencyKey, correlationId, requestId, paymentReference (in record/metadata) |
| `payment_transactions` | transactionId, paymentReference, providerReference + metadata.{correlationId, requestId, traceId, idempotencyKey} |
| `payment_audit_logs` | correlationId, requestId (top-level) + metadata.{traceId, transactionId, paymentReference, providerReference} |
| Event Bus envelope | All six identifiers when available |

**Audit schema note:** Module 3 stores `correlationId` and `requestId` as first-class fields. `traceId`, `transactionId`, `paymentReference`, and `providerReference` are carried in `metadata` until schema extension (optional Module 9 enhancement). Query via `metadata.transactionId` index can be added if query volume warrants.

## 8. Complete Trace Lifecycle

### Phase A вҖ” Request ingress

```
T0  Client initiates POST /checkout/charge
    CREATE  requestId        (this HTTP request)
    RESOLVE correlationId    (header or generate)
    RESOLVE traceId          (header or correlationId)
    RESOLVE idempotencyKey   (header, required for mutations)
    OPTIONAL paymentReference (header or payload)
```

### Phase B вҖ” Idempotency gate

```
T1  IdempotencyService.claim(key)
    PERSIST correlationId, requestId, paymentReference on idempotency record
    IF replay вҶ’ RETURN cached result with original transactionId
    IF new   вҶ’ CONTINUE to handler
```

### Phase C вҖ” Transaction birth

```
T2  TransactionService.createTransaction()
    CREATE  transactionId
    COPY    correlationId, requestId, traceId вҶ’ metadata
    COPY    paymentReference вҶ’ top-level field
    providerReference = null
    AUDIT   PAYMENT_CREATED (resourceId = transactionId)
```

### Phase D вҖ” Provider interaction

```
T3  ProviderAdapter.charge()
    SEND    paymentReference, correlationId (as provider metadata)
    RECEIVE providerReference
    AUDIT   PAYMENT_PENDING / PAYMENT_AUTHORIZED
```

### Phase E вҖ” Capture & settlement

```
T4  TransactionService.transitionStatus(CAPTURED)
    SET     providerReference (first time)
    AUDIT   PAYMENT_CAPTURED (before/after status)
    PUBLISH PaymentCaptured event
      envelope: { traceId, correlationId, requestId, transactionId, paymentReference, providerReference }

T5  EventBus вҶ’ AuditEventSubscriber
    AUDIT   (async) if not already recorded synchronously

T6  EventBus вҶ’ WalletEventSubscriber
    WALLET_CREDITED
    AUDIT   WALLET_CREDITED (metadata links transactionId)
```

### Phase F вҖ” Webhook reconciliation (async)

```
T7  POST /webhooks/mtn-momo
    NEW     requestId (webhook HTTP)
    LOOKUP  transaction by providerReference
    INHERIT correlationId, traceId, transactionId, paymentReference
    AUDIT   PAYMENT_SETTLED or PAYMENT_FAILED
```

### Phase G вҖ” Query / reconstruction

```
Operator query:
  findByCorrelationId(correlationId)  вҶ’ full audit timeline
  findByResource(TRANSACTION, txnId)    вҶ’ transaction-scoped events
  transaction.metadata.traceId          вҶ’ join with APM/logs
  providerReference                     вҶ’ webhook + provider support lookup
```

| Header | Required | Set by |
|--------|----------|--------|

## 10. Failure & Retry Scenarios

| Scenario | correlationId | requestId | transactionId | Behavior |
|----------|---------------|-----------|---------------|----------|
| Network retry, same Idempotency-Key | unchanged | **new** | unchanged | Idempotency replay |
| User clicks Pay again (new key) | **new** | **new** | **new** | New transaction |
| Provider timeout, engine retries internally | unchanged | unchanged | unchanged | Same transaction, new audit events |
| Webhook duplicate | inherited | **new** | unchanged | Idempotency on webhook key |
| Partial failure after create | unchanged | unchanged | unchanged | Transaction вҶ’ FAILED, audit trail complete |
